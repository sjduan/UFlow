UFlow DDR/HBM 数据传输流程汇报

本报告总结当前 UFlow 在本机 DDR、NPU HBM、跨 NPU HBM 之间的数据管理与传输流程。 当前核心原则是：UFlow 服务进程持有数据对象的权威视图和可传输地址，client 通过 lease/shareable handle 获得计算侧访问权，官方数据移动统一由 UFlow 产生 TransferEvent。

1. 当前结论

当前 DDR/HBM 传输已经从“client 拿到指针后自己 copy”收敛为“UFlow 服务侧统一执行并记录事件”。这让后续跨介质、跨卡、跨节点路径能被同一套 cost/event 模型管理。

DDR 默认对象 memfd_shared 作为 authoritative DDR DataObject，适合 fd passing、lease 准入和服务化生命周期管理。
DDR-HBM 热路径 pinned staging memfd mmap 与 aclrtMallocHost pinned pool 之间做 CPU copy，再由 ACL 做 H2D/D2H。
HBM-HBM ACL D2D direct 同卡和跨卡均由 daemon 使用 service VA 执行 direct D2D，跨卡已验证 NPU7→NPU6。
同步模型 async-first 同步只是 SubmitTransfer + WaitEvent；每次传输都产生 TransferEvent 和实际路径统计。
关键边界 Client 可以 import/map shareable HBM handle 用于计算，但 client 自行调用 aclrtMemcpy 或 kernel 修改不会被 UFlow 自动感知。正式传输必须走 SubmitTransfer；client 写入后必须用 MarkDirty / MarkReady 或后续 external event/barrier 对齐。

2. 总体架构：Client 接口、UFlow 模块与冷热路径

UFlow 的服务边界分成 client 接入面、服务控制面、placement runtime、传输执行面和介质后端。Client 侧只消费 DataHandle / runtime descriptor / TransferEvent；UFlow 侧负责对象目录、lease、数据状态、路径规划、实际传输和统计回写。

2.0 一图讲清：接口、模块、冷热数据与冷热路径

这张图的讲法 从上到下讲三层：应用只通过统一 API 申请/打开对象并提交传输；UFlow 服务层统一管理对象目录、lease、service VA、传输规划、事件和统计；底部按资源域展开 D2D、D2H/H2D、H2H Hot 与 H2H Cold，分别对应 HBM 热数据、DDR-HBM 快速换入换出、进程间共享热/温数据，以及未来 UB/SSU/DFS 的冷数据扩展。

2.1 Client 侧接口与能力

Allocator / SDK

Create / Open

申请 HBM 或 DDR DataObject，获得 lease、placement、shareable handle 或 memfd descriptor。

Runtime Adapter

Descriptor

把 DataHandle 解析成 PyPTO/Simpler 可消费的 device ptr、child memory 或 mmap view。

Transfer API

Plan / Submit / Wait

只表达数据移动意图，不直接做 copy；同步调用也会产生 TransferEvent。

State API

Dirty / Ready

client 自行写入后显式标记数据状态，避免 UFlow 用未完成数据执行传输。

控制面：对象、lease、规划和事件 HBM：physical handle + service VA + D2D DDR：memfd/mmap authoritative object pinned pool：默认热传输 buffer

2.2 UFlow 内部数据管理与传输模块

Object 管理

Catalog / Namespace / Lease

记录 DataObject、DataPlacement、client lease、allowed range、state、dirty range 和生命周期。

HBM 管理

Physical + service VA

使用 physical HBM、shareable handle、daemon-side VMM map，支持 H2D/D2H/D2D。

DDR 管理

memfd / mmap

默认 authoritative DDR object 是 memfd_shared；daemon 和 client 分别 mmap，同一份数据由 lease 控制准入。

Pinned 管理

Transfer pool

预分配/复用 aclrtMallocHost pinned chunk，用于高吞吐 DDR↔HBM 热路径。

Transfer 管理

Plan / Execute / Event

根据 DataPlacement 和 cost 选择 direct、pinned staging、D2D、DDR memcpy 或未来 UB/SSU 路径。

Policy 管理

Hot/cold + fallback

热数据优先 HBM/near DDR；热路径优先 pinned/D2D；冷数据走 DDR/SSD/DFS/SSU，按 cost 预取。

3. HBM 数据生命周期与访问流程

3.1 HBM 创建与服务侧 VA

1
CreateDataObject(hint=mandatory:hbm)

client 只表达逻辑对象、size、target npu；不直接申请 device pointer。

2
aclrtMallocPhysical

UFlow daemon 在目标 NPU 上申请 physical HBM handle。

3
Export shareable handle

daemon 导出 shareable token，用于授权 client 计算侧访问。

4
Reserve/Map/MemSetAccess

daemon 立即把同一段 physical HBM map 成自己的 service_device_ptr，并设置 READWRITE 权限。

5
保存 PlacementRuntime

catalog 保存 public metadata；runtime map 保存 raw handle、service VA、device id、mapped bytes。

3.2 Client 访问权

1
Open / lease

client 注册 PID/bare TGID，UFlow 记录 lease 和访问范围。

2
aclrtMemSetPidToShareableHandle

daemon 给目标进程授权 shareable handle。

3
client import/map

client 获得自己的 device VA，可传给 kernel 或 runtime descriptor。

4
计算侧写入后同步

client 修改数据后必须显式 MarkDirty / MarkReady，否则 UFlow transfer 会拒绝或视为未定义。

已验证事实 PhaseE-02 已证明 daemon service VA 与 client import VA 指向同一份 physical HBM；client 写入后通过 MarkDirty + MarkReady，daemon 后续 HBM→DDR 可读到修改。not-ready source 会被拒绝并产生 failed event。

4. DDR 数据生命周期与默认选型

DDR 不再简单等同于普通 host tensor。当前将 DDR 分成 authoritative shared object、pinned transfer pool、direct fallback、registered/hugepage/NUMA opt-in 四类语义。

DDR 形态 角色 当前决策
memfd_shared authoritative DDR DataObject 默认 共享语义清楚，不暴露 path，适合 fd passing 和 lease。
aclrtMallocHost pinned transfer pool 默认热传输 性能最高，但非独立进程共享对象。
direct shared DDR no-bounce fallback 备选 稳定可用，吞吐低于 pinned staging。
registered mmap experimental / opt-in 可用但非默认 V1 register 成功，16MiB register 约 10ms，当前无吞吐优势。
tmpfs_mmap_shared debug/simple backend 保留 便于调试和早期部署。

6. DDR ↔ HBM 默认热路径

为什么默认走 pinned staging

PhaseE-03 100KiB-512MiB sweep 里，64MiB 以上 pinned staging H2D/D2H 约 12-15 GiB/s；direct shared DDR 多数约 3.7-6.6 GiB/s。虽然多一次 CPU copy，但整体更快。

为什么 DDR object 仍用 memfd

pinned host memory 不天然可跨独立进程共享；memfd 的生命周期和授权更适合服务化对象，因此 authoritative storage 与 transfer pool 分层。

为什么 registered mmap 非默认

aclrtHostRegister V1 能注册 mmap VA；register 不计入热路径后，registered memfd 在大对象上仍未超过 memfd+pinned staging。

7. HBM ↔ HBM direct 路径

PhaseE-02 之后，同卡和跨卡 HBM-HBM 都不再依赖 host relay。UFlow daemon 持有 src/dst 的 service VA，传输由 daemon 在正确 device/context 下执行。

8. 状态、同步和事件模型

Data State

Created / Ready / Dirty / Released 是 transfer 消费源数据的前置约束。not-ready source 会产生 failed event。

Dirty / Ready

client 计算侧写 HBM 或 DDR 后，需要调用 MarkDirty 标注修改范围，再调用 MarkReady 允许 daemon transfer 消费。

Async-first

SubmitTransfer 总是创建 TransferEvent；同步调用只是提交后等待。event 回写实际 path、engine、latency、bandwidth 和 fallback reason。

10. 决策树

不要把 pinned host 当成 DDR DataObject pinned host memory 是当前最快的 Host↔Device DMA buffer，但它不是天然跨进程共享对象。UFlow 的正式设计是“共享对象”和“传输池”分层，而不是把一个内存形态强行承担两种语义。

11. 后续接入方向

memfd fd passing把 PhaseE-03 选型纳入 daemon catalog 和 RPC，client 通过 lease 获取 fd。
pinned pool policy按 size、并发、NUMA/TopologyDomain 调整 chunk size 和复用策略。
PyPTO event 对齐TransferEvent 对齐 PyPTO DAG dependency，支持外部 event wait/record。
Cost feedback把 actual bandwidth/latency 回写 cost model，动态选择 direct 或 staging。
UB/remote backend把同一套 DataPlacement/TransferPlan 推广到 remote DDR、SSU、DFS 和 UB shmem。
