Data Service 架构规划报告

面向 950/960 代际超节点和 UB 链路的数据管理服务：把 DDR、SSD、HBM(NPU) 的位置、共享、迁移、同步与异步访问统一成可编程数据平面，同时给 pypto-serving 和其它进程开放手动数据控制接口。

1. 核心范围

Data Service 不是单纯的本机 allocator，也不是重复 910 时代已有的数据管理系统；它面向 950 之后的 UB-native 机器假设，把“数据在哪里、怎么访问、怎么移动、什么时候同步”从 pypto-serving 和 PyPTO runtime 中独立出来。

地址与位置DataObject、DataPlacement、DataHandle 统一描述逻辑数据和物理副本。
多介质本地/远端 DDR、SSD、HBM(NPU) 纳入同一资源模型。
数据流动migrate、replicate、prefetch、sync、flush、invalidate 都走统一接口。
UB 验证通过 ub-sim 验证 remote DDR、remote SSD、HBM direct/relay 语义。

2. 代际背景：为什么是 950 以后

统一数据管理不是新概念。910 / a2a3 代际已经有成熟的内存池、缓存、KV 管理、文件系统和显式搬运方案，但这些方案通常建立在协议和地址空间分段的机器假设上。950 的 UBL128/SU/SO/UBG/Urma/SSU 让这个边界发生变化：数据可以在逻辑上被统一寻址、直接读写、异步完成，并由驱动或底层 runtime 选择物理路径。

12. 意义与创新性

现实意义

  • 把 950 UBL128 中 SU/SO/DCN 的网络职责固化到数据服务策略，避免热路径走错网络。
  • 把 SSU LBA-direct、UB shmem、Urma read/write、SDMA completion 统一成可观察的 TransferPlan。
  • 让 pypto-serving 只表达权重、KVCache、prefix 和 policy，不再嵌入具体搬运路径。
  • 让调度器能按数据位置和迁移成本选择 prefill/decode 节点。

创新点

  • 从“内存池管理”升级为“跨介质、跨节点、跨 UB 域的数据路径编排”。
  • 用 DataHandle 屏蔽本地 ptr、device ptr、UB import、SSU LBA、relay path 的差异。
  • 把异步硬件完成事件纳入数据生命周期，避免 copy 未完成就释放或消费。
  • 通过 ub-sim 在没有完整硬件前验证拓扑、能力、故障和路径选择。

13. 读者结论

Data Service 的第一版应同时满足两个方向：向下能接管当前 Qwen3 权重/KV 的 NPU 内存，向上能暴露迁移、同步、共享、预取等显式接口。这样当前服务接入不会重，未来超节点 DDR/SSD/HBM 打平也有自然演进路径。
