UFlow Overall Architecture

结构版总图：第一张图只保留主模块和跨模块接口流；后续 section 逐个展开模块内部结构。目标是先形成整体感，再按模块下钻细节。

0. Legend

后续所有图统一使用这些线型。主图只画跨模块接口线，模块内部流向放到对应 subsection。

1. Main Architecture

主图只保留 UFlow 的主模块。每条线都表示模块边界上的接口流，不展开模块内部调用。
主图刻意不画 core 内部的细线。它只表达：client 通过统一 API 进入 daemon core；core 负责对象、计划和事件；executor 负责官方数据传输；backend 和 media 承担具体设备/介质能力。

2. Client & API Boundary

client 侧只表达意图：申请对象、提交传输、等待事件、查看状态。是否走 HBM、DDR、fallback、direct，由 UFlow 服务侧决定。

3. Daemon Core Services

核心层只做状态、规则和计划，不直接暴露 ACL 细节。所有对象状态和传输事件都在这里被统一管理。

4. Transfer Executor

这里展开“数据怎么走”。主路径是 direct async，pinned chunk lane 是 fallback/对照能力，事件统一回到 core。

5. Backend Runtime

backend 层负责把统一对象和传输请求落到具体介质、ACL runtime 和 OS memory 能力上。

6. DataObject & Placement

UFlow 的统一抽象是 DataObject。不同介质只是 placement/backend 不同；client 拿到的是 lease/reference，不是传输所有权。

7. Observability: Monitor + Trace

monitor 是常开低频状态；trace 是按需打开的细粒度事件。两者统一从 daemon 输出，不和数据面耦合。
